# LOOP — Build Prompt

## What You Are Building

LOOP is a text-based investigation game set in a five-day time loop at a summer camp. The player relives the same five days (up to 7 loops) trying to prevent a catastrophe that happens on Day 5. Each loop, the player has 20 time slots (5 days × 4 slots: Morning, Afternoon, Evening, Night) to explore locations, talk to NPCs, observe events, search for clues, and piece together the mystery. Knowledge persists between loops; everything else resets.

The game runs in the terminal. It uses rich text effects (colors, Unicode box-drawing, ANSI formatting) but is fundamentally text-based. All NPC dialogue is generated by Claude via the Agent SDK.

Build this project in the directory: `/mnt/c/python/text_game/loop/`

---

## Technology Stack

- **Language**: Python 3.10+
- **LLM Integration**: Claude Agent SDK (`claude_agent_sdk`)
- **Authentication**: Max subscription OAuth (automatic via `~/.claude/.credentials.json`)
- **Terminal UI**: `rich` library for formatted output (panels, tables, colored text, progress bars, markdown)
- **Data**: JSON files for world state, player knowledge base, and save/load
- **No external database. No web server. Single-player, local terminal application.**

### CRITICAL: Authentication Setup

This project uses Claude Max subscription OAuth, NOT an API key. Every module that initializes the Agent SDK must include this guard at the top:

```python
import os
if "ANTHROPIC_API_KEY" in os.environ:
    del os.environ["ANTHROPIC_API_KEY"]
```

Authentication flows automatically through `~/.claude/.credentials.json`. Do NOT set `ANTHROPIC_API_KEY` anywhere — not in code, not in `.env`, not in environment. The SDK will pick up the OAuth token automatically.

Use model `claude-sonnet-4-20250514` for all LLM calls (conversations, world generation). This balances quality and speed for an interactive game.

## Core Architecture

### The Iron Rule: LLM Is the Actor, State Machine Is the Director

The LLM generates all dialogue and narrative flavor. The state machine determines all game consequences. The LLM NEVER decides:
- Where characters are (schedules are deterministic)
- What evidence exists or where it is (set by world generation)
- Whether interventions succeed (set by the intervention tree)
- Game outcomes (set by evaluation of player actions vs. causal chain)

The LLM ALWAYS handles:
- NPC dialogue in conversations
- Narrative descriptions of locations and observations
- Emotional reactions to player questions
- Expressing the current game state through character behavior

### Data Flow Per Player Turn

```
1. Player selects action (talk/observe/search/follow/intervene)
2. State machine validates action and resolves deterministic effects
3. If conversation: build system prompt from character profile + timeline position + trust + player knowledge → send to Claude → stream response
4. State machine extracts structured data from interaction (what was learned, trust changes)
5. Update knowledge base with any new evidence
6. Advance time slot
7. Redraw UI
```

## World Generation (game start)

At the start of a new game, make a single large LLM call that generates the entire world state as structured JSON. This world state is immutable for the playthrough.

### What to Generate

**Camp Geography** — Use a fixed set of locations (these do NOT need to be generated, use the canonical set below):

| Area | Locations |
|------|-----------|
| Main Camp | Lodge, Cabins (8), Fire Pit, Counselor Quarters |
| The Lake | Dock, Swimming Area, Boathouse, Canoe Storage |
| The Woods | Hiking Trail, Ropes Course, Old Amphitheater, Old Camp Ruins |
| Utility | Maintenance Shed, Director's Office, Infirmary, Supply Cabin |

Each location has properties: `name`, `description`, `locked` (bool + conditions), `search_items` (list of discoverable evidence), `adjacent_locations` (for travel).

**The Catastrophe** — Generate one. It must be:
- Dramatic but non-violent to individuals (fire, flood, structural collapse, contamination)
- Caused by human action/negligence (not random nature)
- Preventable only through understanding the full causal chain
- Connected to the camp's history
- Require intervention at multiple points across 5 days to prevent

**The Causal Chain** — A sequence of 5-8 events across the 5 days that lead to the catastrophe. Each event has: `day`, `time_slot`, `character`, `action`, `location`, `is_interruptible` (bool), `interrupt_method`, `downstream_effects`.

**Characters** — Generate 12-15 named characters organized by investigation tier:

| Tier | Count | Role |
|------|-------|------|
| Tier 1: Causal Chain | 3-4 | Directly responsible for the catastrophe |
| Tier 2: Witnesses | 3-4 | Observed something critical without understanding it |
| Tier 3: Red Herrings | 2-3 | Hiding unrelated secrets that look suspicious |
| Tier 4: Atmosphere | 3-4 | Make the camp alive, occasionally useful |

Each character needs:
- `name`, `age`, `role` (camper/counselor/staff), `tier`
- `personality` (3-5 traits), `speech_pattern` (how they talk — formal, slang, nervous, etc.)
- `schedule`: array of 20 entries (5 days × 4 slots), each with `location` and `activity`
- `secrets`: what they're hiding (tier 1-3 characters)
- `knowledge_timeline`: what they know at each point in the 5 days (so the LLM knows what a character can reveal at Day 2 Morning vs Day 4 Night)
- `trust_threshold`: how much trust is needed before they share sensitive info
- `relationships`: attitudes toward other named characters

**Evidence Registry** — Every discoverable piece of evidence:
- `id`, `type` (testimony/observation/physical/behavioral), `description`
- `location` or `character` (where/from whom it's found)
- `available_day`, `available_slot` (when it becomes findable)
- `prerequisites` (other evidence or trust levels needed first)
- `significance` (what it reveals about the causal chain)
- `connects_to` (other evidence IDs it links to)

**Intervention Tree** — For each interruptible event in the causal chain:
- What the player must know to attempt the intervention
- What action they must take (and where/when)
- What changes in the schedule/causal chain if successful
- Cascading effects on other characters and events

**Wild Cards** — 2-3 events that happen regardless of player action at fixed times (e.g., a thunderstorm on Day 3 Evening that changes everyone's location, a surprise visitor on Day 4).

**Ending Conditions** — Specific criteria for each ending type:
- Full Prevention: which causal chain events must be interrupted
- Partial Prevention: minimum interruptions for damage reduction
- Exposure: which evidence must be assembled
- The Deeper Truth: what historical connection must be uncovered

### World Generation Prompt Strategy

Send a single prompt that asks Claude to generate all of the above as a JSON object. Provide the exact JSON schema you expect. Validate the output for:
- Schedule consistency (no character in two places at once)
- Solvability (evidence chain is complete — no dead ends)
- Causal chain coherence (events follow logically)
- All locations referenced in schedules exist
- All evidence prerequisites are achievable

If validation fails, regenerate. Store the validated world state as `data/world_state.json`.

## The Clockwork Engine (state_machine.py)

The clockwork runs in two modes:

**Default Mode** (no player intervention this loop): Characters follow their generated schedules exactly. Events happen on cue. The catastrophe occurs on Day 5 as generated.

**Reactive Mode** (player has intervened): When the player takes an `intervene` action, the state machine:
1. Checks the intervention tree for valid interventions at this day/slot/location
2. If valid, marks the causal chain event as interrupted
3. Recalculates downstream schedules for affected characters
4. Propagates cascading effects through the intervention tree
5. All non-affected schedules remain unchanged

### State Tracked Per Loop

```python
@dataclass
class LoopState:
    loop_number: int              # 1-7
    current_day: int              # 1-5
    current_slot: TimeSlot        # MORNING, AFTERNOON, EVENING, NIGHT
    player_location: str          # current location ID
    character_trust: dict[str, int]     # per-character trust (0-100), resets each loop
    interventions_made: list[str]       # intervention IDs completed this loop
    schedule_modifications: dict        # character schedule overrides from interventions
    events_triggered: list[str]         # wild card and causal events that have fired
    conversations_this_loop: dict       # log of who player talked to and key topics
```

### State Tracked Across Loops (in KnowledgeBase)

```python
@dataclass
class PersistentKnowledge:
    evidence_discovered: dict[str, Evidence]  # evidence ID → evidence data
    character_schedules_known: dict[str, list[ScheduleEntry]]  # what the player knows about routines
    evidence_connections: list[tuple[str, str]]  # player-drawn connections between evidence
    theories: list[str]                          # player-written theory notes
    loop_history: list[LoopSummary]              # summary of each completed loop
    total_actions_taken: int
    characters_met: set[str]
    locations_searched: set[str]
```

## Player Actions

Each time slot, the player picks ONE action:

### Talk To [Character]
- Character must be present at the player's current location this day/slot
- Launches a conversation via the LLM conversation engine
- Player types freely; character responds in-character
- Player types `[leave]` or `[done]` to end conversation
- After conversation, the state machine extracts: topics discussed, evidence mentioned, trust change
- Trust building: +1 to +5 per positive exchange (empathy, remembering details, appropriate questions). Trust loss: -2 to -10 for aggression, accusations without evidence, ignoring emotional cues.

### Observe
- Returns a narrative description of the location: who's present, what's happening, ambient details
- May reveal scheduled deviations happening at this time (someone sneaking somewhere, a whispered conversation)
- Less targeted than talking but can catch things conversations miss
- Generated by LLM from the current location state + who's there + what events are occurring

### Search [Location]
- Only works if the player is unobserved (location is empty or only contains unaware characters)
- If observed, the search is interrupted and the observer's suspicion of the player increases
- Returns discoverable evidence at this location for this day/slot (checking the evidence registry)
- Some evidence requires prerequisites (other evidence found first, or a key/combination known)

### Follow [Character]
- Character must be present and about to leave
- Consumes the NEXT time slot as well (costs 2 slots total)
- Reveals where the character goes and what they do in the next slot
- Detection chance: 20% base, +20% if the character is Tier 1 (cautious), -10% if player trust with them is high
- If detected: trust penalty, and the character may alter their schedule for the rest of this loop

### Intervene
- Only available when the player has sufficient knowledge (specific evidence prerequisites met)
- Present the player with available interventions based on what they know and where/when they are
- Resolve via the intervention tree
- May require a conversation component (convincing someone) which is LLM-driven with a trust/evidence check

### Travel
- Move to an adjacent location. Does NOT consume a time slot unless traveling to a distant location (Woods → Main Camp = costs a slot)
- Adjacent location travel is free and happens before the action

## Conversation Engine (conversation_engine.py)

Each NPC conversation is a Claude call with a carefully constructed system prompt.

### System Prompt Structure

```
You are {character.name}, a {character.age}-year-old {character.role} at Camp Pinehaven.

PERSONALITY: {character.personality}
SPEECH PATTERN: {character.speech_pattern}

CURRENT STATE:
- It is Day {day}, {time_slot}
- You are at {location} doing {activity}
- Your mood is {current_mood}

WHAT YOU KNOW RIGHT NOW:
{character.knowledge_timeline[day][slot]}

YOUR SECRETS:
{character.secrets — include instructions on when/how to reveal}

YOUR TRUST TOWARD THE PLAYER: {trust_score}/100
- Below 20: You are dismissive and uncooperative
- 20-40: You are polite but guarded
- 40-60: You are friendly and somewhat open
- 60-80: You are trusting and willing to share sensitive information
- Above 80: You confide freely

THE PLAYER HAS DEMONSTRATED KNOWLEDGE OF:
{list of things the player knows from previous loops that this character would find surprising}

CRITICAL RULES:
1. Stay in character. You are a real person at summer camp, not an AI.
2. Only share information consistent with your knowledge at this point in the timeline.
3. If the player mentions something you don't know yet or shouldn't know, react with genuine surprise. Ask "How do you know that?" Don't just go along with it.
4. If the player asks about your secrets, guard them until trust is sufficient. Deflect, change the subject, get uncomfortable — don't just refuse robotically.
5. Your responses should be 1-4 sentences typically. This is a conversation, not a monologue. Be natural.
6. Express emotions through behavior, not labels. Don't say "I feel nervous." Say "I... look, can we talk about something else?"
7. If the player is kind and remembers details about you, warm up. If they're pushy or aggressive, shut down.
8. Never break the fourth wall. Never reference being in a game, being an AI, or time loops.
```

### Conversation Flow

1. Display a brief scene-setting description (who's here, what they're doing)
2. Player types their message
3. Build the system prompt + conversation history → send to Claude
4. Stream the response to the terminal with character-appropriate styling
5. After each exchange, run a lightweight trust assessment:
   - Did the player ask about the character's feelings? (+1 trust)
   - Did the player remember a detail the character mentioned earlier? (+2 trust)
   - Did the player demonstrate impossible knowledge respectfully? (+3 trust if explained, -2 if creepy)
   - Did the player accuse or pressure aggressively? (-3 to -8 trust)
6. Repeat until player types `[leave]` / `[done]` or the time slot naturally ends
7. Extract key information learned and add to knowledge base

### Handling "Impossible Knowledge" (cross-loop knowledge)

This is the game's signature mechanic. When the player says something they shouldn't know:

Include in the system prompt a section called `PLAYER KNOWLEDGE FLAGS` listing specific facts the player has learned in previous loops. Tag each with how the character would react:

```
- PLAYER KNOWS: Your sister is sick (you mention this on Day 4 Evening)
  → REACTION: Shocked. "Wait — how do you know about Lily? I haven't told anyone here."

- PLAYER KNOWS: You go to the maintenance shed at Night on Day 3
  → REACTION: Defensive. "Were you following me? That's none of your business."
```

The LLM handles the emotional nuance of these moments. The state machine flags which knowledge items to include.

## Game Flow

### New Game

1. Display title screen with atmospheric ASCII art
2. Run world generation → show a "Camp is waking up..." loading screen while generating
3. Validate world state
4. Save world state to `data/world_state.json`
5. Initialize loop 1, day 1, morning
6. Show opening narrative: player arrives at camp, gets assigned a cabin, meets a couple characters
7. Begin gameplay loop

### Main Gameplay Loop

```
while loop_number <= 7:
    while not (day == 5 and slot == NIGHT and catastrophe_triggered):
        display_location_scene()
        action = get_player_action()
        resolve_action(action)

        if action consumed time slot:
            advance_time()

        check_for_scheduled_events()
        check_for_wild_cards()

        if day == 5 and catastrophe_conditions_met():
            trigger_catastrophe_scene()
            break

    if catastrophe_prevented():
        run_ending_sequence()
        break

    show_loop_reset_sequence()
    show_between_loop_screen()  # review knowledge base, plan next loop
    reset_loop_state()
    loop_number += 1

if loop_number > 7:
    run_failure_ending()

run_post_game_summary()
```

### Between Loops

After each loop reset, give the player a planning screen:
- Review their knowledge base and evidence board
- Pin/unpin characters on the schedule tracker
- Write theories and notes
- Plan their approach for the next loop (this is just for the player — no mechanical effect)
- Option to save game

### Token Management

- World generation: one large call (~4000 output tokens). Happens once.
- NPC conversations: stream responses, keep conversation history within a scene, clear between scenes. Typical scene: 10-20 exchanges.
- Descriptions: short calls (~200 output tokens each). Frequent but cheap.
- Include a token-aware conversation trimmer that keeps the most recent 20 exchanges in context and summarizes earlier ones.

## Endings in Detail

### Full Prevention
Player has interrupted enough causal chain events. Day 5 plays out differently — no catastrophe. An epilogue scene shows the aftermath: investigations, accountability, the camp's future. Generated by LLM from the world state and player's specific interventions.

### Partial Prevention
The catastrophe happens but is reduced in severity. Damage but no danger to people. The player gets an epilogue acknowledging their partial success.

### Exposure
The catastrophe happens in full, but the player has assembled enough evidence. A post-catastrophe scene where the player can present evidence to authorities. The responsible parties are held accountable.

### Failure
The catastrophe happens. The player doesn't have enough evidence or interventions. The loop breaks — time moves forward. A somber epilogue.

### The Deeper Truth (hidden ending)
The player has uncovered the camp's historical connection — the catastrophe is part of a pattern. Preventing it AND exposing the systemic problem leads to the most complete resolution. This requires finding evidence in the Old Camp Ruins and connecting it to the current events.

## Key Implementation Notes

### What to Build First (priority order)

1. **Models + Config** — Define all data structures. Get this right first.
2. **World Generator** — Generate a test world state. Validate it manually. Iterate on the generation prompt until worlds are consistently good.
3. **State Machine** — The clockwork engine. Test it independently: advance time, check schedules, verify characters are where they should be.
4. **Display** — Basic terminal UI. Show location, time, available actions. Get the game loop feeling right with placeholder text.
5. **Conversation Engine** — NPC dialogue. This is where the game comes alive. Test with one character extensively before expanding.
6. **Knowledge Base + Evidence Board** — Persistence across loops. The player's memory system.
7. **Actions** — All five action types fully implemented.
8. **Interventions** — The reactive clockwork. How player actions change the world.
9. **Loop Reset + Between-Loop Screen** — The meta-game layer.
10. **Endings + Post-Game** — Evaluation and summary.
11. **Polish** — Map, schedule tracker, visual effects, title screen.

### Conversation Quality is Everything

The NPCs need to feel like people. Spend significant effort on the conversation system prompts. Test these scenarios specifically:
- Player asks a character about something they shouldn't know yet → surprise/suspicion
- Player is kind and remembers details → character warms up
- Player is pushy and aggressive → character shuts down
- Player presents evidence to a suspect → defensive, scared, or confessional depending on trust
- Player has a casual non-investigation conversation → character has hobbies, opinions, feelings

### Difficulty Calibration

- Loop 1-2: Player is exploring, learning the camp, meeting people. Information comes easily. The schedule is a blank slate filling in.
- Loop 3-4: Player is testing theories, following suspects, searching locked locations. Information is gated behind trust and prerequisites.
- Loop 5-6: Player is executing plans, making interventions, dealing with cascading consequences. The clockwork is reactive.
- Loop 7: Emergency. If the player hasn't solved it by now, this is their last chance. The UI should communicate urgency.

### Save System

Save the complete game state (world state + knowledge base + current loop state) as a JSON file. Support multiple save slots. Save automatically between loops. Allow manual saves at any action prompt.

### Error Handling

- If an LLM call fails, retry once. If it fails again, use a fallback response ("The person seems distracted and doesn't respond" / "You don't notice anything unusual").
- If world generation produces invalid JSON, retry with a more constrained prompt.
- Never crash the game on an LLM error. Degrade gracefully.

## What Success Looks Like

A 12-year-old sits down, starts the game, and within 10 minutes is exploring the camp, talking to characters, and feeling the pull of the mystery. By loop 2, they're scribbling notes (in-game or on real paper). By loop 3, they're planning their time slots like a chess player. By loop 5, they're executing a plan that makes them feel like a genius. When the post-game summary reveals everything they missed, they immediately want to play again.

The conversations feel real. The camp feels real. The clock feels relentless. The moment they use knowledge from a future loop and watch an NPC's reaction — that's the moment the game has them.
